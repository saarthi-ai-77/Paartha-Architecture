# SOS-001: State Ownership, Lifecycle, and Boundary Specification

**Role note:** produced in parallel with EXP-019, per CTO directive, to formalize the ownership, lifecycle, and boundaries of every internal state object in ACA — not to add new state, and not to resolve open questions this document did not itself set out to test. Where a boundary depends on EXP-019's still-pending result, that entry is marked **PENDING** rather than guessed at.

**Why this document exists now, not earlier:** every prior architecture document (ARS-001, ACA v1.0, IVS-001, DAS-001, CTX-001) named pieces of this picture piecemeal, as each was needed for a specific decision — S_episodic's three original content schemas (ACA v0.4 §1.1), IVS-001's key-namespacing patch for routing, EXP-018's discovery that a fourth schema (context) interacts badly with the shared eviction policy, CTX-001's proposal of a fifth schema ("unknown"). **No single document has ever catalogued all of them together, with one consistent ownership/lifecycle/boundary format.** IVS-001's Stage 3/4 minimal integrated prototype has not yet been executed; this document exists so that when it is, the state model it integrates against is unambiguous, not reconstructed from five different documents' partial treatments.

---

## 1. What Counts as State Here

Persistent state: anything that outlives a single inference call and can affect a later, independent call. This explicitly **excludes** the ephemeral intermediate representations that flow between COMPOSE → EVALUATE → SELECT *within* one forward pass (ARS-001 §5.2: COMPOSE's output is "ephemeral... not intended for further structural composition" beyond the current computation; EVALUATE and SELECT similarly consume and produce values that do not, by themselves, persist). Only UPDATE has a required persistent side effect (ARS-001 §5.2). This document catalogs only what UPDATE can act on — the state substrates and their content schemas — not the transient values passed between functions during one call.

## 2. Catalog of State Objects

### 2.1 S_episodic (ME-01) — the shared mechanism

Fixed-capacity, content-addressable key-value store (ACA v1.0 §1.1). One write/evict *mechanism* — competence-gated write (entropy/loss above a running threshold), competence-gated eviction (lowest-signal entry when full) — hosts multiple content *schemas*, each with its own key space, write trigger, and eviction semantics. **The mechanism is one substrate; the schemas are not separate substrates, but they are not identical in behavior either — this section exists because that distinction has been informally understood but never written down as a single table.**

| Schema | Key | Value | Write trigger | Evict-eligible? | Evidence |
|---|---|---|---|---|---|
| **Fact** | input representation (e.g. name_id) | correct output | Gated: EVALUATE finds current S_semantic prediction wrong | Yes — ordinary competence-gated eviction | **Validated** (EXP-001), but scope-limited: fails to protect against forgetting under sequential, non-rehearsed training (EXP-018/010) |
| **Routing decision** | input-class signature | chosen COMPOSE family ID | Gated: EVALUATE (held-out performance) prefers a family over the current default | Yes, but **untested whether ordinary eviction incorrectly treats a still-load-bearing routing entry as redundant** (IVS-001 §2, "Routing ↔ Learning" row — a named, unmitigated risk, not yet run as EXP-011/017) | **Reasoned hypothesis** (ACA v0.4 §1.1) |
| **Self-model entry** | domain/skill identifier | aggregated EVALUATE score over recent probes | Periodic, driven by DP-03's probe cycle | **Untested** — no experiment has ever exercised this schema; its eviction behavior under load is a genuine open question, not merely an unlikely one | **Speculative** (ACA v0.4 §1.1) |
| **Reasoning-chain (RC-03)** | step index within the current chain | intermediate result | Every step of an active multi-step composition | Short-horizon by design — evicted once the chain concludes or a bounded horizon is exceeded | **Reasoned hypothesis**, "~90% reducible to ME-01" (ARS-001 §1.4); IVS-001's audit confirmed no separate *machinery* is needed, but did not test *tuning* adequacy |
| **"Unknown" (proposed, not yet adopted)** | flagged input representation | low-confidence marker + context | EVALUATE-LOCAL below threshold ("I don't know") | Proposed to require protection from ordinary fact-eviction pressure until a separate, scheduled review (CTX-001 §3's "Gap Curriculum Generation," offered to DAS-001) — **not yet implemented, no eviction policy specified** | **Reasoned hypothesis** (CTX-001 §3) — this row is a proposal, included here so it is visible in the same table as everything else, not because it is adopted |
| **Context / working state** | slot identifier, in a **reserved capacity partition** ordinary fact-eviction cannot touch | current field value | Every reassignment, **unconditional** (never gated — see Section 4) | **No** — structurally excluded from ordinary fact-eviction's candidate set, by partition, not by chance | **Resolved by EXP-019**: a dedicated substrate is unnecessary; a reserved partition within S_episodic, written unconditionally, performs identically (1.000 ± 0.000 recall, both) to full structural separation. Naive schema-tagging without reservation collapses to 0.200 ± 0.100 with zero coverage — not from eviction-after-entry (EXP-018's mechanism) but from **write starvation**: a saturated store with nothing currently "mastered enough" to evict can permanently lock out a new, high-priority entry. See Section 3-4 below and `docs/06_experiments/Completed.md` (EXP-019). |

### 2.2 Substrates outside S_episodic

**S_semantic (ME-02):** slow-updating trainable parameters — the general backbone and each COMPOSE module's structural parameters (ACA v1.0 §1.2). Owner: UPDATE (ordinary training gradient steps; consolidation via replay, **Reasoned hypothesis, minimal form falsified by EXP-010**). Readers: COMPOSE, and indirectly EVALUATE-LOCAL/GENERALIZATION (which measure S_semantic's current behavior — this is the "competence" measurement CTX-001 §2 formalized as distinct from information storage). Lifecycle: created at initialization, updated continuously by ordinary training, versioned at deployment checkpoints (DP-02). Never evicted — S_semantic has no capacity limit in the same sense S_episodic does; its "forgetting" (EXP-018) is a side effect of continued training, not an eviction decision.

**S_invariants (new in v1.0, §1.3):** read-only, externally-supplied domain invariants. Owner: **nothing in the architecture** — this is the one substrate with no internal writer by design (ACA v1.0 §1.3: "an invariant a system could train itself out of respecting is not serving its purpose"). Readers: EVALUATE-STRUCTURAL only. Lifecycle: supplied at specification time by whoever defines the domain, static thereafter, versioned only when the domain specification itself changes (not a runtime event). **Evidence: Reasoned hypothesis**, contingent on EXP-015.

**2.2a — S_working: resolved, not adopted.** EXP-019 (`docs/06_experiments/Completed.md`) falsified the need for a genuinely separate substrate — a reserved capacity partition within S_episodic, written unconditionally, is sufficient and performs identically to full structural separation. Context/working-state content therefore lives in Section 2.1's table as a disciplined S_episodic partition, not a fourth substrate. Its lifecycle: created at interaction start (or, per EXP-019's design, forced-initialized before any query can occur, so evaluation never tests an untrained value), overwritten unconditionally every relevant turn, and — the one part of this lifecycle EXP-019 did not test — either discarded at interaction end or promoted into the ordinary fact schema (as a specific memory of what happened) or S_semantic (via consolidation, inheriting EXP-010's known fragility). This promotion path is exactly CTX-001 §3's "Gap Curriculum Generation" territory and remains unspecified.

**Deployment artifact (DP-01):** not a live state substrate — a packaging/serialization boundary over the substrates above (ARS-001 §5.6: "orthogonal to the computations themselves"). Must capture S_episodic's contents (or a stated, principled subset), S_semantic's parameters, and S_invariants as a static file (IVS-001 §5). **Speculative** — the least-resolved requirement in the entire specification, unchanged since ACA v0.4.

## 3. The Canonical Key-Namespacing Rule

IVS-001's prerequisite audit patched one instance of this problem ("S_episodic's shared content schemas must be key-namespaced by schema... before EXP-011 runs") without stating a general rule. EXP-019's implementation independently invented a namespace-offset convention for its own purposes (`SLOT_KEY_OFFSET`). **This section makes the rule general and canonical, so the next experiment does not have to reinvent it a third time:**

Every S_episodic schema owns a disjoint key range or key-prefix, fixed at design time, never inferred at runtime:
- Fact keys: raw input-representation identifiers (e.g., name_id), as validated by EXP-001.
- Routing keys: input-class signatures, namespaced distinctly from fact keys (IVS-001's patch).
- Self-model keys: domain/skill identifiers, a third disjoint range.
- Reasoning-chain keys: step indices, scoped to the currently-active chain (reused across chains once a chain concludes — the only schema whose keys are intentionally not globally unique, since chain state is discarded at chain end).
- "Unknown" keys (if adopted): flagged input-representation identifiers, namespaced distinctly from ordinary fact keys even when the flagged input is the same representation a fact entry might otherwise use — schema, not content, determines namespace.
- Context/working-state keys (if living in S_episodic per EXP-019's outcome): slot identifiers, namespaced distinctly from all of the above (EXP-019's own `SLOT_KEY_OFFSET` is a working instance of this general rule, not a one-off).

**This is a Validated-by-design fix, not an empirical claim** — namespace collision is a specification gap, closed by construction, the same status IVS-001 gave its own routing patch.

## 4. The Write-Gating Rule — a Distinction This Program Had Not Previously Made Explicit

EXP-019's design surfaced a real, useful distinction that no prior document stated in general terms: **not every S_episodic schema should use the same write trigger.**

- **Gated write** (write only if EVALUATE finds current competence insufficient — EXP-001's validated mechanism): correct for schemas where the content is one of *many equally-valid, independent* items competing for bounded capacity, and where writing something the backbone already handles wastes a slot (facts, routing decisions, self-model entries, reasoning-chain steps, "unknown" flags).
- **Unconditional write** (always overwrite the current value, regardless of surprise): correct for schemas where the content is *the single current value of a small, fixed field*, and where declining to record a reassignment because it isn't "surprising enough" would leave a stale value in place — exactly wrong for a field whose entire purpose is tracking what is currently true. Context/working-state content is the one schema confirmed to need this (EXP-019).

**Confirmed by EXP-019, not merely proposed:** the naive condition used gated writes for context (treating it like a fact), and this was the proximate mechanism of its failure — not primarily via eviction, but via **write starvation** (Section 2.1): a saturated, gated store persistently found nothing "mastered enough" to evict, so genuinely-surprising context writes were locked out entirely, in every seed, at every one of 24 events. Unconditional write plus reserved capacity together closed this gap completely (1.000 ± 0.000 recall).

## 5. Known Unresolved Boundaries (Named, Not Papered Over)

- **Self-model schema eviction behavior is completely untested.** No experiment has exercised this schema at all. Its interaction with ordinary fact-eviction pressure is an open question of exactly the same shape EXP-018 answered for facts and EXP-019 is answering for context — not yet asked.
- **"Unknown" schema's interaction with ordinary eviction is untested** (CTX-001 §3 proposed it; this document is the first to note it inherits the same open risk category as every other schema sharing the mechanism, until its own eviction policy is actually specified and tested).
- **Cross-timescale version skew** (DAS-001 §9): restated here as a state-boundary concern, not just a deployment concern — S_semantic and the routing schema within S_episodic update on independent schedules, and nothing in this catalog resolves what happens when they drift out of sync. Still unmitigated.
- **Reasoning-chain (RC-03) vs. context (working-state partition) boundary**, per CTX-001 §4: these sit on the same spectrum (short-horizon working state) but at different points — RC-03's original motivating case is discrete, step-indexed results; context is a small number of continuously-overwritten fields. This document treats them as two rows in Section 2.1 on CTX-001's argument. EXP-019 validated that context specifically needs a reserved partition + unconditional write; whether RC-03's discrete, step-indexed case also needs this same discipline (rather than ordinary gated writes, which may suit it fine given its naturally key-indexed shape) is **untested** and should not be assumed answered by EXP-019 — a distinct, still-open question.
- **Write-starvation risk for other schemas.** EXP-019 named a general mechanism (Section 4): any gated, capacity-constrained schema can starve a high-priority write if the store is persistently saturated with other still-surprising content. This was tested for context; it has **not** been tested for the routing or self-model schemas, both of which use the same gated-write mechanism and could in principle exhibit the same failure. Flagged as follow-up research in EXP-019's own write-up, not yet scheduled.
