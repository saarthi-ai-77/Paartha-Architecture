# CTX-001: The Computational Nature of Context, Cognitive State, and Learning

**Role note:** produced in the same reduction-first, falsification-first discipline as `docs/08_requirements/ARS-001.md` Sections 5 and 7, applied to a new candidate boundary rather than a new architecture layer. **The objective of this document is not to invent a new architectural component.** It is to determine whether memory (EXP-018/EXP-010), conversation context, and "known unknown" tracking are independent problems, or symptoms of one underspecified abstraction — and if the latter, to characterize that abstraction precisely enough to be falsifiable, not to design it.

**Trigger:** a CTO directive observing that EXP-018/EXP-010 (information preservation ≠ competence preservation), the proposed unknown-handling pipeline, and the standing question of how conversation context should be represented, "appear, at first glance, unrelated" but may not be independent. This document investigates that suspicion and attempts to falsify it before accepting it.

---

## 1. Method: the Same Admission Criterion ARS-001 Already Uses

`docs/08_requirements/ARS-001.md` §5.1 admits a new function only if it has (a) a clear, typed input/output signature, (b) at least two plausible, genuinely different implementations, (c) demonstrated necessity traced to specific requirements, not asserted. This document applies the identical bar to any candidate new *substrate* or *function*, and additionally requires: **a reduction attempt must be shown to fail, not skipped.** Every candidate below is first tested against the four existing functions (EVALUATE, SELECT, UPDATE, COMPOSE — ACA v1.0 §2) and three existing substrates (S_episodic, S_semantic, S_invariants — ACA v1.0 §1). Only where a genuine reduction attempt fails is anything new proposed, and even then it is proposed as a **Reasoned Hypothesis** with a named falsification test, not adopted.

## 2. Reduction Attempt 1 — Competence vs. Information (the Best-Evidenced Case)

**Claim under test:** "knowledge is preserved if information is preserved" (the assumption ME-03/EXP-001 implicitly relied on).

**Reduction attempt:** is "competence" just a property of S_episodic (is the fact still stored)? No — EXP-018 measured `with_memory_acc` (a function of S_episodic's contents) and `param_only_acc` (a function of S_semantic's current behavior) *separately*, and found them **exactly equal in every reported row**, with S_episodic coverage at exactly **0.0**. That means the entire observed Stage-1 accuracy in every condition — including `competence_gated_memory` — came from S_semantic, not from S_episodic, despite S_episodic being the component EXP-001 validated. Competence is not "information present in a store"; it is **a measurable property of S_semantic's current behavior on a given query, obtained by applying EVALUATE to it** — it was already expressible in existing terms, it was simply not being *tracked separately* from information-storage before EXP-018 forced the distinction.

**Does this reduce cleanly?** Yes. `Competence(query) := EVALUATE(S_semantic's output on query, reference)`. No new function or substrate required — EXP-018/010's contribution is not a new computational primitive, it is the *empirical discovery that two already-distinct quantities had been conflated*, and a corrected vocabulary: **"information" (a fact exists in some store) and "competence" (a query against S_semantic currently succeeds) are different measurements over different substrates, related but not identical, and EXP-010 shows the relationship is not simple ("reinforce S_semantic once" does not reliably raise competence against subsequent interference).**

**Evidence:** EXP-018, EXP-010 (`docs/06_experiments/Completed.md`). **Dependency trace:** ME-03 (`ARS-001.md`), Section 6/11/18 of `ACA_v1.0_Architecture.md`. **Confidence:** **Validated** — this is the one conclusion in this document resting on direct experimental measurement, not analysis alone.

## 3. Reduction Attempt 2 — The "Unknown" Pipeline

**Claim under test:** does "ACA responds 'I don't know' → store → accumulate → find recurring patterns → mentor-guided learning → validate → promote" require a new first-class computational object ("known unknown"), or is it an engineering workflow over existing primitives?

Walking the pipeline stage by stage, against the existing four functions:

| Stage | Reduces to | New machinery needed? |
|---|---|---|
| Detect low confidence ("I don't know") | EVALUATE-LOCAL, thresholded (ACA v1.0 §2.2a) | No — already validated (EXP-009) |
| Store the flagged experience | UPDATE writing to S_episodic, under a third schema ("unknown"), alongside the already-established fact/routing schemas | No — a schema addition, the same kind of change IVS-001's prerequisite audit already made for routing (namespacing facts vs. routing keys) |
| **Accumulate + identify recurring patterns** | Attempted reduction: repeated EVALUATE (pairwise or cluster-similarity scoring) → repeated SELECT (grouping decisions) | **Partial.** The *scoring and choosing* operations are existing functions: this is not a new fundamental function. But nothing in ACA v1.0's specified pipelines (§5, the training/inference/deployment lifecycles) describes a **periodic, scheduled pass that re-examines a growing subset of S_episodic as a batch** — every existing use of EVALUATE/SELECT is per-query, not per-corpus. This is closer to a missing **orchestration timescale** than a missing function — see below. |
| Mentor-guided learning | Already a separately-tracked, separately-evaluated external proposal (`docs/05_research/proposals/developmental_mentor_paradigm.md`, status: "Requires Prerequisite Research") | No — cross-reference, not re-derivation. Not re-litigated here. |
| Validation | EVALUATE-GENERALIZATION, labels-only (ACA v1.0 §2.2b, EXP-009) | No — already specified, already known to require real labels |
| Promotion into semantic knowledge | UPDATE, semantic consolidation (ACA v1.0 §2.4) | No new function — but **this step inherits EXP-010's result directly**: the one form of "reinforce S_semantic from stored content" this program has actually tested (a one-time replay burst) was just falsified. Any unknown-pipeline design that ends with "promote into semantic knowledge" is proposing to lean on a mechanism this program has direct, fresh evidence is fragile, not a solved building block. |

**Conclusion:** the unknown-handling pipeline reduces almost entirely to existing functions and a schema addition to an existing substrate. The one piece that does not fully reduce — periodic cross-corpus pattern detection — does not require a new *function* (it is expressible as repeated EVALUATE+SELECT) or a new *substrate* (it operates over S_episodic's existing "unknown"-schema entries); it requires a new **scheduled orchestration process**, structurally the same kind of gap DAS-001 already found once (a "Routing Revalidation" timescale that the original five-timescale learning proposal omitted, `docs/10_deployment/DAS-001.md` §7). This document tentatively names it **"Gap Curriculum Generation"** as a fifth candidate timescale, sitting between Memory Consolidation and Semantic Retraining in DAS-001's ordering — proposed for that document to adopt or reject, not adopted here.

**Evidence:** structural/analytical (no dedicated experiment yet). **Dependency trace:** ARS-001 ME-03/SR-01, ACA v1.0 §2.2a/§2.4/§7, DAS-001 §7 (timescale model), the mentor-paradigm proposal. **Confidence:** **Reasoned hypothesis** — the reduction argument is sound, but "Gap Curriculum Generation" as a named timescale has no falsification test yet and is not proposed for adoption without one.

## 4. Reduction Attempt 3 — Conversation Context and Reasoning State

**This is not a new concern.** `docs/08_requirements/ARS-001.md` §2 (ME-01 entry, "Open questions") already asked: *"whether the same substrate serves both 'correction learning' and 'short-horizon working state during multi-step reasoning' (RC-03) equally well, or whether these are actually two different usage patterns needing different tuning of the same underlying mechanism."* This was left open at ARS-001's own admission (RC-03 was flagged "~90% reducible to ME-01," `ARS-001.md` §1.4) and later audited by IVS-001, which concluded **"RC-03 has no separate identity — it is S_episodic (ME-01) used with a short-horizon eviction policy, not distinct machinery"** (`IVS-001.md`, prerequisite audit). That audit answered whether RC-03 is *separate machinery* (no). It did not answer — and does not claim to answer — whether *one write/evict policy tuning* actually serves both purposes well. CTX-001 is the first document to examine that specific, still-open residual.

**Attempted reduction:** is "conversation context" (current topic, goal, resolved concepts, open questions, assumptions, user intent) just S_episodic entries under a fourth schema, the same move that worked for routing and for "unknowns" above?

The access pattern does not match. S_episodic (ME-01, ACA v1.0 §1.1) is a **content-addressable, key-indexed store**: insert a (key, value), evict by policy, query by key. Facts, routing decisions, and flagged unknowns are all naturally key-indexed — "for this input class, which family," "for this name, which city," "was this specific query low-confidence." Conversation context, as described (a small, bounded set of fields — topic, goal, resolved-set, open-question-set, assumption-set) is not naturally key-indexed at all: there is no useful "key" for "the current topic" other than "the current topic itself," and the operation performed on it every turn is not insert-and-later-retrieve, it is **overwrite/refine a small fixed-size record given the previous record and one new input** — the exact signature of UPDATE `(State, NewInformation, Gate) → State'` (ARS-001 §5.2), but applied to a state object with a fundamentally different shape and size discipline than S_episodic's (bounded number of *typed fields*, not a growing/evicting number of *keyed entries*).

**This is the one candidate in this document that does not cleanly reduce.** UPDATE as a *function* still applies (nothing new needed there) — what is missing is a fourth *storage target* with different structural properties from all three existing substrates:

| Property | S_episodic | S_semantic | S_invariants | Conversation/reasoning context |
|---|---|---|---|---|
| Size discipline | Bounded count of entries, evict-by-policy | Fixed parameter count | Fixed, externally supplied | Bounded *fields*, no eviction — overwritten in place |
| Update cadence | Gated (surprise/competence) | Slow (training) | Never (read-only) | Every turn/step |
| Indexing | Content-addressable (key → value) | N/A (distributed) | N/A (rule set) | Not naturally keyed — a small structured record |
| Lifecycle | Persistent until evicted | Persistent, slowly drifting | Static | Ephemeral by default; may be promoted (to S_episodic, as "what happened," or to S_semantic, via consolidation) when the process concludes |

None of the three existing substrates has this shape. Tentatively naming this **S_working** — not proposed for adoption, named only so it can be referred to precisely and tested.

**Why this is not merely "RC-03 restated":** RC-03's original motivating case (ARS-001 §1.4, §2) was intermediate results across a reasoning *chain* — naturally sequential, plausibly still key-indexable by step number. Conversation-topic-tracking is a sharper case on the same spectrum: not a growing sequence of discrete step-results, but a small number of fields under continuous revision. CTX-001's contribution is narrowing ARS-001's already-open question from "does ME-01 serve RC-03 well?" to a sharper, testable version: **"does a key-indexed, evict-by-policy store serve a small-fixed-field, continuously-overwritten record well, or does that access-pattern mismatch itself cause degradation as the record's lifetime grows?"**

**Evidence:** none experimental — this is a structural/access-pattern argument, not a measurement. **Dependency trace:** ARS-001 §2 (ME-01 open question, explicitly not resolved by IVS-001's audit), ACA v1.0 §1.1. **Confidence:** **Reasoned hypothesis.** Falsification test proposed in Section 7.

## 5. Central Synthesis — Attempting to Falsify "Cognitive State Is the Primary Object"

**The hypothesis as stated:** ACA currently treats *information* as its primary computational object; the correct primary object is *cognitive state*.

**Attempted falsification of the strong/naive version:** if this were straightforwardly true, simply re-labeling context (and unknowns, and memory) as "cognitive state" should resolve the problems observed. It does not. **A hypothetical S_working substrate would face the exact same retention-under-sequential-pressure problem EXP-018 just demonstrated for S_episodic**, unless it also carries a working update/retention *policy* that survives sequential, non-static conditions — which naming it "cognitive state" does not supply. EXP-010 already shows the obvious first-guess fix (reinforce, then evict) does not solve this even for the simplest existing substrate. **The strong version of the hypothesis is therefore false, or at minimum unsupported: swapping the ontological label from "information" to "cognitive state" is not itself a fix for anything this program has actually observed going wrong.**

**What survives, more narrowly:** the three motivating cases share a common shape, but it is not "they are all secretly cognitive state" — it is that **they all depend on a bounded, actively-maintained representation that must remain useful as the underlying process continues to evolve, and this program has now validated (EXP-001, positively) and then falsified in its simplest fix (EXP-018/EXP-010, negatively) that maintaining such a representation correctly is a real, unsolved, general problem — not a solved prerequisite that a new substrate could simply assume.** "Cognitive state" is at most a reasonable *label* for the state-model gap identified in Section 4 (S_working); it is not evidence that a deeper unifying principle has been found, and this document explicitly declines to claim more than that.

**Confidence:** **Reasoned hypothesis, deliberately narrowed from the directive's original framing.** The appealing, general version of the claim is rejected; the precise, evidenced version is retained and stated as narrowly as the evidence supports.

## 6. Token History vs. Structured Cognitive State — Derived, Not Inherited

**Requirement-driven derivation, not a Transformer-inherited assumption:** a context representation must support (a) unbounded interaction length without unbounded cost, (b) being *queried* for "what currently matters" without recomputing it from scratch each time, (c) cheap incremental update as new input arrives.

Token history satisfies (c) trivially (append) but not (a) (cost grows with history, architecture-dependent but never free) or (b) (extracting "current topic" from raw history requires re-deriving it via attention on every single forward pass — the structured summary is implicit and must be reconstructed, not maintained). A bounded, actively-maintained structured state is the better fit *in principle* for the same reason EXP-001 already validated in a different setting: **EXP-001 showed that "keep everything, don't discriminate" (naive caching) is worse than a policy that actively decides what's worth retaining under a bounded budget.** Token history, treated as context, is structurally the "keep everything" strategy; a maintained structured state is the "actively curate under a budget" strategy — the same class of choice EXP-001 already decided, in a different instance.

**The honest complication, stated rather than suppressed:** EXP-018/EXP-010 showed that *even a discriminating, actively-curated bounded store* (competence-gated memory) fails once the underlying process is sequential rather than static, and its most direct fix does not help either. Conversation context is inherently sequential (every turn is new pressure on the state, exactly like Benchmark A's stages). **The conclusion is therefore not "structured state wins" — it is "structured state is very likely the right kind of object, for the same reason bounded curation already beat raw hoarding once (EXP-001), but this program has no validated retention/update policy that survives sequential pressure for *any* bounded substrate yet — so adopting structured context without solving that shared problem would just relocate EXP-018's open failure into a new component, not avoid it."**

**Evidence:** EXP-001 (analogy, not direct test), EXP-018/EXP-010 (direct, but for S_episodic, not context). **Confidence:** **Reasoned hypothesis** — the direction (structured over raw history) is well-argued; the adequacy of any specific retention policy is untested and should not be assumed.

## 7. Required Analysis — Distinct Entities or Shared Representation?

| Term | Distinct entity, or a view onto something else? | Basis |
|---|---|---|
| Raw information | Distinct — unprocessed (key, value) content, or raw input before any function is applied | Definitional |
| Knowledge | Not distinct — informal name for "information that has survived consolidation and is robustly retrievable/usable"; a *status*, not a substrate | Section 2 |
| Competence | Distinct from information-storage — a measured property of S_semantic's current behavior via EVALUATE, not a stored quantity | Section 2, **Validated** by EXP-018/010 |
| Context / Conversation state | Tentatively distinct — the one case that does not reduce to existing substrates without strain (Section 4) | Section 4, Reasoned hypothesis |
| Learning opportunities / Unknowns | Not distinct — a third schema within S_episodic (like routing before it), plus a not-yet-specified scheduled orchestration process, not a new substrate or function | Section 3 |
| Memory | Umbrella term for S_episodic specifically, in this architecture's existing terminology | ACA v1.0 §1.1 |
| Reasoning state | Same spectrum as conversation context (Section 4) — the discrete, step-indexed end reduces cleanly to RC-03/S_episodic; the continuously-overwritten-summary end does not | Section 4 |

## 9. Proposed Next Experiment — EXP-019 (Not Run Here)

**Falsification test for S_working (Section 4):** build a bounded, multi-turn context-tracking task (e.g., a sequence of queries against an evolving small "current understanding" state, analogous in spirit to Benchmark A's staged structure but for a summary-field record rather than discrete facts). Compare (a) an S_episodic-with-schema-tag implementation (bounded key-value entries tagged "context," evicted by the existing competence-gated policy) against (b) a dedicated small, fixed-field, directly-overwritten state object, both maintained across an extended interaction. **Falsifying outcome for S_working:** if (a) shows no measurable degradation relative to (b) as interaction length grows, the access-pattern-mismatch argument in Section 4 was a plausible-sounding but practically-inert concern, and no new substrate is warranted — RC-03's original "~90% reducible" framing was, in practice, closer to 100%. **Confirming outcome:** if (a) measurably degrades (e.g., "current topic" drifts, gets evicted alongside unrelated facts, or requires disproportionate capacity to protect) while (b) does not, this is real evidence for a structurally distinct fourth substrate, not just a naming preference.
