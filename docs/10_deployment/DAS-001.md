# DAS-001: Deployment & Runtime Architecture Specification

**Scope discipline, stated up front because it governs every section below:** this document does not redesign ACA. Every conclusion traces through Architecture (ACA v1.0) → Function → Requirement (ARS-001) → Evidence. Where the architecture as validated doesn't determine an answer, that is stated as an open question (Section 12), not resolved by invention.

**Headline finding, derived once and then used throughout:** EVALUATE's split into LOCAL and GENERALIZATION (EXP-009) is not merely an evaluation-quality fact — it is a **runtime-determining fact**. ME-03 (memory gating) is gated by EVALUATE-LOCAL, validated label-free (EXP-001, EXP-009): it can run at live inference time, on arbitrary input, with no external label. RC-02 (routing/family-selection) is gated by EVALUATE-GENERALIZATION, validated only with real labels (EXP-002/003) and falsified label-free (EXP-009): **it cannot run at live inference time on arbitrary user input, full stop, unless EXP-015 changes this.** Almost everything below is a consequence of this one asymmetry, not a new design choice.

---

## Answering the Additional Requirement First: Does ACA Require a Fundamentally Different Runtime?

**No — not for anything currently validated.** Everything ACA v1.0 has actually validated (COMPOSE's forward pass, ME-03's write/evict gated by label-free EVALUATE-LOCAL) fits inside a conventional `load → prompt → generate → return` loop without modification. **The things that would require genuinely new runtime machinery — scheduled consolidation, routing revalidation, structural evaluation — are, without exception, still Reasoned Hypothesis, not validated.** It would be unjustified to conclude ACA "requires" a new paradigm on the strength of hypotheses that haven't been tested yet. What can be said honestly: *if* those hypotheses pan out, ACA's full realization adds a **scheduled, off-the-inference-path orchestration layer** alongside a conventional core — closer in kind to "a standard model deployment plus a RAG-style side-index plus an MLOps retraining pipeline" (all of which already exist in production systems) than to any new execution paradigm. This is stated as the headline conclusion, not buried at the end, because the whole document is organized around defending or overturning it.

## 1. Runtime Architecture

Two layers, cleanly separable, derived from the LOCAL/GENERALIZATION asymmetry above:

**Frozen inference core (mandatory, conventional):** S_semantic weights, the COMPOSE library, EVALUATE-LOCAL (entropy — per IVS-001's audit, the required default), a **frozen routing table** (family-selection decisions fixed at the last validation event, not decided live). This is architecturally indistinguishable from deploying any standard neural network — *Evidence: Validated* (every component here traces to EXP-001/002/003/009).

**Scheduled orchestration layer (optional, off the inference path):** episodic memory writes (can run synchronously within the inference loop, since EVALUATE-LOCAL is label-free-validated — this is the one orchestration piece that *can* live inside live inference); memory consolidation (periodic, batch, EXP-010 — *Reasoned hypothesis*); routing revalidation (triggered, requires fresh labeled data, not yet named anywhere before this document but directly implied by EXP-009 — *Reasoned hypothesis*); structural-evaluation-based pre-filtering (EXP-015 — *Reasoned hypothesis*).

## 2. Deployment Artifact

Directly extends IVS-001 §5 rather than re-deriving it. The artifact is a **composite bundle**, not "just weights": (a) S_semantic weights — mandatory, standard; (b) S_invariants specification — mandatory if structural evaluation is used, trivial to include since it's a static, read-only file, never trained (*Validated-by-definition*, per ARS-001 §7); (c) S_episodic snapshot — mandatory per EXP-001's own finding that memory content materially affects performance, with the explicit option to ship empty for a pure frozen-mode deployment; (d) the frozen routing table — mandatory given RC-02's labels-only status. Precedent for "weights plus an index" already exists in production (kNN-LM, RETRO — already cited in this program's own literature review) — this is an extension of known practice, not an invention.

## 3. Lifecycle of an ACA Instance

`Semantic (re)training (once or periodically, off-path) → Routing validation (produces the frozen table, off-path, requires labels) → Artifact assembly (Section 2) → Deployment → Live inference (frozen core + optional synchronous memory writes) → [scheduled: consolidation, revalidation] → re-assembly of a new versioned artifact.` Every named transition already exists in ACA v1.0 or IVS-001 (Learning Pipeline, memory pipeline, Section 12's roadmap) — this section sequences them, it does not add new ones.

## 4. Learning Lifecycle — Three Timescales, Not One, Derived Not Assumed

The directive proposed five stages (Runtime / Experience Collection / Memory Consolidation / Semantic Learning / Architecture Evolution). Checked against the architecture, not accepted as given:

- **Runtime and Experience Collection are the same process, not two.** Episodic writes are a normal part of the inference loop's operation (gated by EVALUATE-LOCAL), not a separate downstream stage — merged.
- **Memory Consolidation and Semantic Learning are legitimately different**, but not in the way named: *Semantic Learning* is the one-time (or rarely-repeated) pre-deployment training of COMPOSE modules (ACA v1.0 §5, step 1 — *Validated*, EXP-002); *Memory Consolidation* is the ongoing, post-deployment trickle of episodic content into those same weights (EXP-010 — *Reasoned hypothesis*). Same mechanism (UPDATE), different lifecycle phase.
- **A timescale missing from the proposed five, but architecturally necessary: Routing Revalidation.** Directly implied by RC-02's labels-only status (EXP-009) — routing cannot update online, so it must update on its own scheduled/triggered cycle, distinct from consolidation.
- **Architecture Evolution is real but is not part of the deployed runtime at all** — it is the human/research process that produced v0.1→v1.0 (this entire session), operating on a timescale of research decisions, not deployed-system cycles.

**Net result: four timescales survive scrutiny, not five** — Runtime (merged), Consolidation, Routing Revalidation (added), Semantic Retraining, with Architecture Evolution explicitly outside the runtime boundary. This is presented as a correction derived from evidence, per the standing rule that architecture yields to evidence, not the reverse.

**Measurable advantage of separating them (Q4's actual question — not assumed):** timescale separation provides a genuine, derivable **poisoning-resistance gradient**. A single adversarial interaction can influence episodic memory directly (Fast timescale) but only ever contributes as one diluted sample among many at Consolidation, and cannot touch routing at all without independent access to the labeled revalidation pipeline. This is a measurable, structural property, not a vague appeal to "separation is good practice" — *Reasoned hypothesis*, not yet empirically measured, but a specific, falsifiable claim (see Section 9).

## 5. Memory Lifecycle

Unchanged from ACA v1.0/IVS-001: write (gated by EVALUATE-LOCAL, *Validated*) → evict (gated the same way, *Validated*) → periodic consolidation (*Reasoned hypothesis*, EXP-010). Nothing new added here — this section exists in DAS-001 only to confirm the memory lifecycle was already fully specified and needs no runtime-specific extension.

## 6. Runtime State Model

At any moment, a deployed ACA instance's full state is: `(S_semantic weights, S_episodic contents, S_invariants file [static], frozen routing table, semantic-version identifier, routing-table-version identifier)`. The last two are new relative to ACA v1.0 — required by Section 8's versioning discussion, not invented independently of it.

## 7. Deployment Modes

Derived directly from measured (not assumed) compute cost, drawing on this session's own experience running EXP-001–009 on a 6GB laptop GPU — **explicitly caveated: this is toy-scale evidence only; cost at real SLM/LLM scale is unmeasured and is named as an open question (Section 12), not extrapolated.**

- **Local/workstation, consumer GPU:** frozen core mandatory and cheap (standard NN forward pass); episodic memory adds negligible cost (bounded by capacity `C`, a lookup, not a training step) — both fully supportable.
- **Edge:** the frozen core alone, with S_episodic/UPDATE disabled entirely, has the **same resource profile as any comparable frozen NN** — continual learning is opt-in at the architecture level, not mandatory, so edge deployment is not precluded by anything validated. Ensemble-based EVALUATE-LOCAL (K× cost) and structural evaluation of unknown cost should default to **excluded** on edge targets pending measurement.
- **Cloud inference clusters:** can support the full frozen core plus synchronous memory writes plus, if validated, structural pre-filtering.
- **Distributed / scheduled infrastructure:** consolidation and routing revalidation belong here by nature (periodic, batch, off the inference path) — they never need to run on the same hardware as live inference at all.

**Mandatory everywhere:** S_semantic + COMPOSE + EVALUATE-LOCAL-via-entropy. **Optional, target-dependent:** S_episodic/UPDATE, ensemble/self-assessment EVALUATE-LOCAL variants, structural evaluation. **Never edge-appropriate, never blocking edge deployment either:** consolidation, revalidation.

## 8. Version Management

Extends IVS-001's DP-02 (already *Reasoned hypothesis*) with a specific mechanism, derived from LN-02 and from ACA's own institutional principle (`shared/FOUNDATIONAL_PRINCIPLES.md` P-006: decisions and rejected hypotheses remain permanently traceable) applied to a running system rather than only to documents. **Semantic weights and the routing table must be versioned independently** (`semantic_v1.2`, `routing_v1.2` as separate identifiers), because they change on different schedules (Section 4). Rollback is well-defined only if consolidation writes are batch-isolated per LN-02 — an unconstrained, continuously-drifting system has no clean rollback point. *Reasoned hypothesis throughout* — no experiment has yet run more than one training/consolidation cycle to test this.

## 9. Security Implications

- **Episodic memory poisoning (new, not previously named anywhere in this program):** because EVALUATE-LOCAL gates writes on the model's *own* assessment of surprise, an adversary could craft inputs specifically designed to appear surprising, causing incorrect content to be written into memory and served back on future similar queries. This is a real attack surface a frozen-weights-only system does not have; it has a known analogue in the RAG-poisoning literature. **Mitigation is exactly the timescale-separation property derived in Section 4:** a single interaction's influence is bounded to episodic memory alone and is diluted, not directly injected, at consolidation.
- **Consolidation-stage poisoning:** requires poisoning *many* episodic entries that then get sampled together — a higher bar than the fast-timescale attack, a genuine derived safety property of batching, not an assumption.
- **Routing poisoning:** lowest risk of the three — requires access to the labeled revalidation pipeline itself, not just crafted inference-time input, precisely *because* EXP-009 already proved routing can't be steered by clever inputs alone.
- **S_invariants as a security asset, not just a function input:** being read-only and externally supplied, it cannot be corrupted by any runtime interaction — a positive security property that falls directly out of how it was already defined (ARS-001 §7), not added here.
- Standard model-extraction / IP concerns apply as they would to any deployed NN — not ACA-specific, noted for completeness only.

## 10. Resource Requirements

Stated honestly rather than fabricated: only toy-scale costs have been measured (this session's own experiments, laptop-GPU, seconds-to-minutes per run). **Real resource requirements at SLM/LLM scale are unmeasured** and are not extrapolated here — this is the same standing scale-transfer gap already tracked in `docs/03_foundations/OPEN.md`, not a new one. Qualitatively: frozen-core cost scales as any comparable NN's would; episodic memory adds `O(capacity C)` lookup cost, cheap by construction; consolidation/revalidation are periodic batch-training-scale costs, off the critical path, and thus do not need to be provisioned alongside live inference hardware at all.

## 11. Failure Modes

Inherits IVS-001's integration risks (memory↔routing feedback loop, routing-eviction regression, calibration drift, capacity contention) without repeating them — see IVS-001 §2 directly. **One new failure mode specific to the runtime/lifecycle analysis in this document: cross-timescale version skew.** Because consolidation (medium timescale) and routing revalidation (slow timescale) run independently, S_semantic can be updated by a consolidation cycle while the frozen routing table remains validated against the *previous* semantic version — family-selection accuracy established pre-consolidation may not hold post-consolidation, silently. **Mitigation, stated as a requirement, not yet implemented or tested:** either trigger routing revalidation automatically whenever a consolidation cycle completes, or periodically spot-check routing validity against the current semantic version between full revalidations. *Reasoned hypothesis* — this risk was not identifiable before Section 4 worked out that the two timescales are independent; it is a genuine finding of this document, not inherited from prior work.

## 12. Open Research Questions

- Real-scale resource measurement (Section 10) — standing, not new.
- Whether the poisoning-resistance gradient (Section 4/9) is actually measurable, not just structurally plausible — needs its own experiment, not yet named with an ID.
- Whether cross-timescale version skew (Section 11) actually manifests, and whether the proposed mitigation is sufficient — untested.
- Whether structural evaluation's runtime cost (Section 7) is cheap or expensive — entirely unknown until EXP-015 specifies concrete checks.
- DP-01's core packaging question remains open exactly as IVS-001 left it — this document sharpens what must go in the bundle (Section 2) but does not resolve the outstanding "how" of packaging it.

## 13. Scientific Confidence Summary

| Conclusion | Confidence |
|---|---|
| Frozen core fits conventional deployment unmodified | **Validated** (EXP-001/002/003/009, directly) |
| Routing cannot run label-free at live inference time | **Validated** (EXP-009's direct falsification) |
| Episodic writes can run synchronously, safely, at inference time | **Validated** for the gating mechanism (EXP-001/009); **Reasoned hypothesis** for "safely" under sustained integration — this is exactly what IVS-001's EXP-016 is designed to test, cited not re-derived |
| Four (not five) learning timescales, with Routing Revalidation added | **Reasoned hypothesis** — a derived correction, not yet empirically tested as a deployed system |
| Poisoning-resistance gradient across timescales | **Reasoned hypothesis** — structurally plausible, not measured |
| Semantic/routing independent versioning is necessary | **Reasoned hypothesis** — derived from LN-02 and P-006, untested |
| Edge deployment viable in frozen mode | **Reasoned hypothesis**, informed by toy-scale measurement only |
| Cross-timescale version skew is a real risk | **Reasoned hypothesis** — newly derived, unmitigated, untested |
| ACA is architecturally (A) a neural network or (B) a cognitive runtime containing one | **Both, precisely scoped:** (A) for the frozen inference core alone; (B) for the complete system including memory and the slower timescales. Asking for one answer across all deployment configurations is underspecified — the dependency on configuration *is* the answer, derived directly from Section 7's decomposition, not a hedge. |
