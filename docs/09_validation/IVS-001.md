# IVS-001: Integrated Validation Strategy for ACA v1.0

**Role note:** this document is produced in a validation role, not a design role. The objective is not to improve ACA v1.0 — it is to determine whether ACA v1.0 survives contact with implementation. The default assumption throughout is that it does not, until a specific test says otherwise.

**Prerequisite audit (required by the directive that produced this document, performed before anything below was written):** RC-03 has no separate identity — it is S_episodic (ME-01) used with a short-horizon eviction policy, not distinct machinery, and is not treated as a component anywhere below. Of the three EXP-009-validated EVALUATE-LOCAL realizations, entropy becomes the required default (free, tied for best on both tested uses); ensemble and the self-assessment head demote to documented fallbacks, not co-required components. Variable compute-amount SELECT is excluded from the minimal prototype — no experiment has ever shown hard-argmax-only is insufficient. A specification gap is patched here rather than deferred to an experiment: **S_episodic's shared content schemas (facts, routing, self-model) must be key-namespaced by schema** before EXP-011 runs, since without it a routing key could collide with a fact key — this is a foreseeable correctness gap, not an empirical uncertainty. One open contingency to check, not decided now: if EXP-015 shows structural pre-filtering fully resolves family selection, the labeled EVALUATE-GENERALIZATION step for that decision becomes redundant and should be removed, not retained as belt-and-suspenders.

## 1. System-Level Success Criteria

Validation is conjunctive, not averaged: ACA is not "validated" if it scores well on average across criteria while failing one. All must hold **simultaneously**, on the same running system, not on separately-run components.

| ID | Criterion | Measurement | Why this is architecture-level, not component-level |
|---|---|---|---|
| SC-1 | Memory retains its tail-recall advantage when embedded in the full pipeline | Tail-fact recall accuracy, integrated system vs. EXP-001's isolated result, same seeds | EXP-001 validated this in isolation only |
| SC-2 | Routing/family-selection accuracy matches EXP-003's isolated result once embedded | Held-out accuracy, integrated vs. isolated | Interaction with memory/consolidation could silently degrade the labeled selection split's integrity |
| SC-3 | Continual learning remains stable over an extended run | Tail-recall and routing accuracy measured at multiple checkpoints across a run far longer than any prior experiment, not just at one endpoint | No experiment to date (EXP-001–009) has measured anything beyond a single fixed-length run evaluated once at the end |
| SC-4 | Evaluation remains calibrated while the system continues to learn | Wrongness-detection AUC re-measured at multiple checkpoints during an extended run, not only on a static post-training snapshot | EXP-009 validated calibration on a frozen model; concurrent learning during use was never tested |
| SC-5 | Multi-step (planning-adjacent) chaining does not degrade long-horizon memory | Tail-recall accuracy before vs. after a chaining-heavy episode, on the shared S_episodic substrate | EXP-004 tested composability under disjoint parameters specifically, avoiding this exact interaction |
| SC-6 | No catastrophic cross-module interaction | No single integration risk in Section 2 manifests at a severity that breaks another module's validated behavior | This is the explicit target of Section 2, not assumed by default |

## 2. Integration Risks — System Interaction Matrix

| Interaction | Expected behavior | Failure mode | Observable symptoms | Recovery strategy | Confidence |
|---|---|---|---|---|---|
| **Memory ↔ Evaluation** | EVALUATE-LOCAL's confidence correctly gates ME-03's write/evict even as S_semantic keeps updating | Calibration drift: as S_semantic changes (ordinary training or consolidation), the confidence signal's correlation with true correctness could decay silently | Tail-recall degrading over a long run relative to a short one — never measured before | Scheduled recalibration checks (periodic small-labeled-holdout AUC re-test) | Reasoned hypothesis — plausible, untested |
| **Evaluation ↔ Routing** | Structural pre-filter plus labeled selection correctly gate family choice | A new, previously-unnamed feedback risk: could a memory "correction" write-back ever leak into what effectively functions as training signal for routing, reinforcing a wrong choice in a loop? | Routing accuracy worse embedded than EXP-003's isolated 0.750 | Keep the labeled selection split architecturally firewalled from anything UPDATE touches — testable | Reasoned hypothesis — this specific loop was not previously identified anywhere in ARS-001/v1.0 |
| **Planning ↔ Memory** | Short-horizon chaining state and long-horizon facts coexist in S_episodic without contention | Capacity contention: a deep or long chaining episode could evict genuinely-important long-horizon facts, or vice versa | Tail-recall dropping specifically during/after chaining-heavy episodes | Either reserve per-schema capacity sub-allocations, or verify empirically that competence-based eviction handles contention gracefully without partitioning | **High-confidence predicted risk** — the shared-substrate design was already flagged as untested for this exact interaction |
| **Routing ↔ Learning (UPDATE)** | Routing decisions persist as long as they remain load-bearing | A routing entry could be evicted as "mastered" (low current discrepancy) while still being the *only* record of which family handles a class — silent regression in routing for previously-well-handled input, indistinguishable from normal fact eviction under the current shared policy | Routing accuracy for a previously-correct input class degrading with no new data suggesting it should | Distinguish "redundant because the backbone learned it" from "still load-bearing infrastructure" — these need different eviction semantics, which the current shared-schema policy does not yet provide | High-confidence predicted risk — directly sharpens what EXP-011 must actually check (not just point-in-time accuracy, but regression over time) |
| **Memory ↔ Routing (key space)** | Facts and routing entries coexist in one substrate without cross-contamination | Key-namespace collision between content schemas sharing one physical key space | A routing entry retrieved as if it were a fact, or vice versa | **Patched in this document's prerequisite audit**, not deferred: namespace keys by schema before EXP-011 | Validated-by-design fix — a foreseeable gap, not an empirical uncertainty |
| **Learning ↔ Deployment** | Deployment artifact captures what's needed to continue validated behavior post-deployment | Untestable as stated — DP-01 is undefined | N/A | See Section 5 | N/A — this interaction cannot be assessed until Section 5's requirements are specified |

## 3. Minimal Integrated Prototype

**Goal: validation, not capability.** Reuses ~90% existing, already-validated code rather than building anything new, and adds only what's needed to exercise the specific risks in Section 2 — nothing added for generality's sake.

**Composition:**
1. EXP-004's joint model (recall pathway + rule pathway, disjoint parameters) as the base.
2. Replace EXP-004's explicit non-learned dispatch with routing decisions stored **in S_episodic** (testing EXP-011), with schema-namespaced keys (this document's audit fix applied).
3. Insert a structural self-composition-consistency pre-filter (testing EXP-015) ahead of the existing labeled family-selection step, on the same rule-task world already used by EXP-002/003.
4. Use entropy alone as EVALUATE-LOCAL (per the audit's pruning decision) — no ensemble, no self-assessment head, unless Stage 3 results demand reconsidering that.
5. Extend the run duration well beyond any prior experiment's step count, with checkpointed evaluation at multiple points (not just the end) — the cheapest possible way to exercise SC-3/SC-4.
6. A trivial chaining stress-test: force a short sequence of repeated recall-pathway queries where each query's input is a function of the previous output, to exercise the Planning↔Memory capacity-contention risk without building any real planning/search module.

This single prototype exercises every risk row in Section 2 and every criterion in Section 1, using components that are already built, at the cost of a few hundred lines of glue code — not a new system.

## 4. Validation Ladder

**Stage 1 — Individual mechanisms. ✅ Complete.** EXP-001 (memory), EXP-002/003 (compose + select), EXP-009 (evaluate-local). *Why Stage 2 is safe:* each has a clean, reproducible, isolated result *and* an understood failure mode from its own falsification history — nothing is being combined blind.

**Stage 2 — Pairwise interactions.** EXP-004 (memory × rule pathway, disjoint parameters) done; EXP-011 (memory × routing), EXP-015 (evaluation × routing via structural filter, tested standalone first), and the planning × memory contention check (new — folded into Section 3's prototype rather than a separate experiment, since it's cheap to include directly) remain. *Why Stage 3 is safe once these pass:* pairwise testing is designed to catch two-way interference, the dominant failure term in most systems — **stated honestly as an assumption, not a guarantee: higher-order (3+-way) interactions could still exist that pairwise testing misses**, and Stage 3's simultaneous-criteria requirement (Section 1) is exactly the check that would catch what pairwise testing alone cannot.

**Stage 3 — Subsystem integration.** The minimal integrated prototype (Section 3), evaluated against *all* of Section 1's criteria simultaneously. *Why Stage 4 is safe once this passes:* Stage 3 already combines every component pairwise-cleared in Stage 2 under extended, checkpointed conditions — Stage 4 differs mainly in adding the still-unvalidated pieces (consolidation, novel-input fallback) rather than in kind of integration.

**Stage 4 — Complete architecture.** Full v1.0, including EXP-010 (consolidation) and EXP-012 (novel-input fallback), still on synthetic tasks. Real-backbone/real-data integration is explicitly a *separate, later* step beyond Stage 4, not part of it — conflating "complete architecture validated on synthetic tasks" with "validated at scale on real data" would be exactly the kind of overclaim this document exists to prevent.

## 5. Deployment Artifact (DP-01) — Requirements, Not a Solution

**Properties any deployment artifact must satisfy:**
- Must capture S_episodic's contents, or make a principled, stated decision to discard specific parts of it — **Validated constraint**, since EXP-001 already showed memory contents materially affect measured performance; discarding it is not a neutral default.
- Must capture S_semantic's parameters — **Validated-by-convention** (standard ML practice; uncontested).
- Must capture routing state as a first-class concern *if and only if* EXP-011 shows routing needs treatment distinct from ordinary facts — **Reasoned, contingent constraint.**
- Does **not** need to capture S_invariants as trained/learned state — it is read-only, externally supplied, and ships as a static specification file alongside the artifact, not extracted from a trained system — **Validated-by-definition**, following directly from S_invariants' own definition, no experiment required.
- Should **not** need to preserve ephemeral short-horizon (chaining) state across a deployment boundary — a fresh instance restarting mid-chain doesn't need to resume someone else's abandoned reasoning — **Reasoned constraint.**
- Must support versioning/rollback (DP-02) — **Reasoned constraint**, following from LN-02.

**Requirements that depend on DP-01:** ME-01, ME-02, RC-02 (routing state, contingently), SR-01 (if a self-model is externalized). **Does not depend on it:** S_invariants (ships separately, statically), ephemeral chaining state (legitimately resets).

**Candidate mechanisms and what would falsify each — not solving DP-01, characterizing it:**
- **Candidate A — full snapshot** (weights + entire episodic store + invariants file). *Falsifying test:* does artifact size grow unboundedly over a long run? Given ME-01's fixed capacity `C`, this is *predicted* to stay bounded — a candidate this document expects to survive, not guaranteed to.
- **Candidate B — weights only, discard episodic memory at the deployment boundary.** *Falsifying test:* does discarding S_episodic cause a measurable tail-recall regression at deployment time? Given EXP-001's own result, this is *predicted to fail immediately* — stated as a prediction now, not deferred to a mystery.
- **Candidate C — consolidated/compressed summary instead of raw episodic store.** *Falsifying test:* entirely gated on EXP-010 — if consolidation-via-replay doesn't work, this candidate has no mechanism to attempt and is not viable until EXP-010 resolves.

## 6. Experimental Dependency Graph — Recomputed by Blast Radius

Ranked by how much of the architecture changes if the experiment fails — **not** by how novel or how previously-planned the experiment is.

| Rank | Experiment | If it fails, what changes |
|---|---|---|
| 1 | **EXP-016 (new): extended-duration calibration-drift test** | If EVALUATE-LOCAL's calibration degrades under concurrent learning, this invalidates something *already validated and relied upon everywhere* (ME-03, SR-01/02) — a bigger architectural shock than a new capability failing to pan out, because it retracts existing infrastructure rather than declining to add new infrastructure |
| 2 | **EXP-015: structural evaluation** | S_invariants, EVALUATE-STRUCTURAL, and the routing pre-filter design are removed entirely if this fails — large, but *new*, not previously relied-upon, machinery |
| 3 | **EXP-011 (basic) + EXP-017 (new: longitudinal routing regression, folded into the same test)** | If routing-as-episodic-content fails even at a point-in-time accuracy check, the "one substrate, shared schema" economy is reversed for routing specifically; if it passes that but fails the longitudinal regression check, the eviction policy needs schema-aware semantics, a smaller fix |
| 4 | **EXP-013: other label-free generalization proxies** | Lower than EXP-015 — extends an already-falsified approach rather than testing a qualitatively new one |
| 5 | **EXP-014: RC-04 classification** | Reshapes Section 8-of-ACA-v1.0's planning design specifically, not the rest of the architecture |
| 6 | **EXP-010: consolidation via replay** | Affects long-run memory capacity management and deployment Candidate C; does not affect correctness of anything already validated |
| 7 | **EXP-012: novel-input fallback** | Matters more as domain coverage grows; least urgent for a first, narrow validation pass |

## 7. Kill Criteria

- **KC-1 (fundamental redesign):** if EXP-016 shows EVALUATE-LOCAL's calibration collapses toward chance under realistic continual-learning conditions, with no mitigation restoring it — the single most load-bearing validated function fails under integration, forcing a fundamental redesign of the evaluation subsystem, not a patch.
- **KC-2 (scope reduction, not death):** if neither EXP-015 nor EXP-013 ever produces a working label-free proxy, and real labels are genuinely unavailable in a target setting — autonomous family selection must be abandoned for that setting; ACA re-scopes to settings where labeled/curated routing remains available. This does not kill ACA; it kills a specific ambition for it.
- **KC-3 (significant but non-fatal redesign):** if Stage 2 pairwise interactions resist all reasonable mitigation (namespacing, capacity partitioning, isolation) — the shared-substrate economy is abandoned in favor of separate substrates per content type. The four-function core and three-way EVALUATE split would likely survive; only this specific simplification would not.
- **KC-4 (most severe — return to Stage 1/2 research):** if Stage 3's minimal prototype fails to satisfy Section 1's criteria simultaneously, for reasons **not** attributable to any risk named in Section 2 — genuinely emergent, unanticipated failure. This means the validation program itself had a blind spot, and warrants returning to component-level research rather than continuing to refine v1.0's current structure.
- **KC-5 (standing, not failure-triggered):** if a simpler architecture (fewer functions, fewer substrates) is ever shown to satisfy the same validated requirements equally well, v1.0 should be simplified to match. This is a standing pressure toward minimality, not a response to breakage.

## 8. Revision Policy

**Experiments modify architecture; architecture never modifies experiments; evidence always overrides design** — mechanically:
1. Every experiment is logged in `docs/06_experiments/Completed.md`, regardless of outcome.
2. If it validates a component, that component's evidence tag in ACA v1.0 is updated in place (a tag change, not a rewritten description).
3. If it falsifies a component, the component moves to ACA v1.0's falsified-beliefs table (§11) with its replacement — the live section is only then modified, never edited invisibly.
4. No live architecture section may imply stronger evidence than the latest experiment supports; this document's own audits (as performed at the top of this file) are the periodic check for that drift.
5. IVS-001 itself follows the same rule: as validation-ladder stages complete or kill criteria trigger, this document is extended with new entries, not rewritten in place.

## 9. EXP-018 (ACA-MVP-001 Benchmark A) — Section 2 Risk Confirmed, Broader Than Scoped; Section 6 Ranking Needs Revision

Per Section 8's own revision policy, recorded as a new entry rather than edited into Section 2's or Section 6's existing rows.

**What happened:** ACA-MVP-001 Benchmark A tested ME-03 (competence-gated memory) against a real Transformer under staged, non-rehearsed continual training — the first test of this mechanism outside EXP-001's static-distribution setting, and independently the first real-Transformer test this validation strategy anticipated (Stage 3/4 of Section 4's ladder, run earlier and narrower than the full minimal integrated prototype in Section 3, but directly relevant to it). Result: the pre-registered 2x-improvement criterion failed cleanly; memory coverage of the oldest facts was exactly 0.0 at evaluation time in every seed, both memory conditions. Full account: `docs/06_experiments/Completed.md` (EXP-018).

**Section 2 update:** the "Routing ↔ Learning (UPDATE)" row predicted that a routing entry could be evicted as "mastered" while still load-bearing. EXP-018 confirms this exact mechanism empirically — for ordinary factual content, not routing. **The risk as written is confirmed but under-scoped**: it is a property of point-in-time-mastery-based eviction under sequential training in general, not something specific to routing entries. Any future revision of Section 2 should generalize this row rather than treat it as routing-only.

**Section 6 (blast-radius ranking) needs revision, not yet performed here:** Rank 6 (EXP-010, consolidation via replay) was justified by "does not affect correctness of anything already validated." EXP-018 shows ME-03 — validated by EXP-001, relied upon throughout this document (Sections 3, 5) — does not survive the one realistic condition (sequential, non-rehearsed staged training) it had not yet been tested under, and EXP-010 is the most direct candidate fix. This raises EXP-010's actual blast radius: it may determine whether ME-03's validated status extends past a static distribution at all, not merely whether long-run memory capacity management improves. **This document is not re-ranking Section 6's table in place here** — consistent with Section 8's own rule, that revision should happen as its own dated entry once EXP-010 is actually run and its outcome is known, not preemptively based on motivation alone.

**Relationship to SC-1/SC-3/SC-4 (Section 1):** EXP-018 is a real, if narrow and non-checkpointed, data point on SC-1 (memory's tail-recall advantage embedded vs. isolated) — and the data point is negative, under this specific staged setting. It is explicitly **not** a test of SC-3 or SC-4 (both require checkpointed measurement across an extended run; EXP-018 measured only stage boundaries and a final endpoint) and should not be cited as satisfying either.
